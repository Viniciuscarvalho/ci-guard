## Pass A — Python project tests vs testing-boss doctrine

Each row maps a testing-boss rule to a concrete check in the ci-guard test suite.
`PASS` means the code satisfies the rule.
`GAP` means it violates or does not implement the rule.
`PARTIAL` means partial compliance with a documented residue.

## Pass B — ci-guard skill doctrine vs testing-boss doctrine

Checks whether `~/.claude/skills/ci-guard/SKILL.md` + references implement testing-boss's flake and CI discipline. Scope: doctrine alignment, not the Python code.

| #   | Rule (testing-boss)                                               | Verdict     | Evidence                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| --- | ----------------------------------------------------------------- | ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| B1  | Flaky discipline: named owner + fix-by date                       | **GAP**     | testing-boss: "named owner within 24h, fix-by date" for quarantined tests. ci-guard ledger schema (`ledger.py:75-86`): fields are `first_seen`, `last_failure`, `last_pass`, `failure_count_30d`, `pass_count_30d`, `flake_rate`, `status`, `quarantined_at`, `quarantine_issue_url`, `events`. No `owner`, `fix_by`, or `reason` fields. Ownership lives in the linked GitHub issue (indirect), but the ledger itself cannot answer "who owns this quarantine?" or "when must it be resolved?"                                            |
| B2  | Anti-pattern #21 — quarantine-as-cemetery                         | **PARTIAL** | ci-guard issue URL tracking: `quarantine_issue_url` field links to a ticket (PARTIAL compliance). But: `cmd_prune` explicitly skips quarantined entries (`ledger.py:236`); `flaky-detection.md` confirms "Quarantined entries are never pruned." There is no system-enforced SLA, no automated staleness alert, no re-entry gate. A quarantine is permanent until manual `set-status --status fixed`. testing-boss requires "named owner, ticket, SLA, and re-entry gate (e.g., ten consecutive clean runs)" — only the ticket is present. |
| B3  | flaky_rate SLO < 1–2%, alert at > 5%                              | **PARTIAL** | ci-guard quarantine threshold: `QUARANTINE_RATE_THRESHOLD = 0.05` (5%) aligns exactly with testing-boss's alert level. Per-test `flake_rate` is computed and surfaced in `ledger query` and `quarantine-candidates`. But: no `ci-guard report` command aggregates a repo-level flaky_rate; SKILL.md (`v0.4.0`) does not cite the < 1–2% / > 5% SLO thresholds from testing-boss; no automated build alert fires when a test's rate exceeds 5%.                                                                                             |
| B4  | Iron Law #3 — fix production first; write why when changing tests | **PARTIAL** | For `branch_failure`, ci-guard SKILL.md Gate 2 says "Never retry. Patch code." — consistent with Iron Law #3. For `test_flake`, the skill routes to retry + verify, not to modify the test. But `record_failure` (`ledger.py:103`) accepts no `reason` argument. When an agent accepts a test as flaky and quarantines it, there is no ledger field capturing why the test is accepted as flaky rather than fixed. The ledger records _what_ happened; not _why_ the decision was made.                                                    |
| B5  | Retry budgets vs Iron Law #4 (real systems gate)                  | **PASS**    | ci-guard's default budgets (2/job, 5/PR, 90 min/PR in `config.py`) restrict retries rather than replacing real execution. The `verify_flaky_green` protocol (SKILL.md Gate 4) requires two consecutive greens on the same SHA — this IS an implementation of Iron Law #4: real execution validates a green, not the trust of a single pass. Budgets and verification reinforce the law.                                                                                                                                                    |
| B6  | AI-gate — agents writing tests must commit to the lowest layer    | **PARTIAL** | ci-guard SKILL.md explicitly excludes test authoring: "Do NOT use this skill for writing new tests or fixing tests." This is correct scope discipline. But when Gate 2 routes an agent to "patch code" for a `branch_failure`, the skill does not cross-reference testing-boss's layer guidance (Iron Law #2, Pattern #10 lowest-layer placement). An agent exiting ci-guard to fix a failing test has no bridge to testing-boss's layer doctrine.                                                                                         |
| B7  | SKILL.md version vs package version (bonus check)                 | **GAP**     | `SKILL.md` frontmatter: `version: 0.4.0`. Package (`pyproject.toml:7`): `version = "0.5.0"`. SKILL.md commands still reference pre-v0.5 script paths: `python3 .ci-guard/scripts/ci_watch.py`, `python3 .ci-guard/scripts/flaky_ledger.py` (lines 51, 115, 148). After v0.5 the unified CLI is `ci-guard watch`, `ci-guard ledger`, etc. Agents following the SKILL.md instructions will fail on v0.5 installs.                                                                                                                            |

## Follow-ups

Ordered by risk (highest-impact or easiest to fix first):

1. **[CRITICAL] Sync SKILL.md to v0.5 CLI** — Update `~/.claude/skills/ci-guard/SKILL.md` command examples from `python3 .ci-guard/scripts/ci_watch.py ...` to `ci-guard watch ...`, `ci-guard ledger ...`, etc. Bump `version:` to `0.5.0`. Agents following the current SKILL.md break on every v0.5 install. (addresses B7)

2. **[HIGH] Add `owner`, `fix_by`, `reason` to ledger schema** — Extend `_ensure_test()` in `ledger.py` with optional `owner`, `fix_by`, `reason` fields. Make `fix_by` defaultable (+14 days) and `reason` required when calling `set-status --status quarantined`. Update `flaky-detection.md` to document these fields and their SLO meaning. (addresses B1, B2, B4)

3. **[HIGH] Surface stale quarantines** — Add a `ci-guard ledger stale-quarantines` subcommand that lists quarantined entries where `fix_by` is past (once field exists). Document the `flake_rate SLO < 1–2%, alert > 5%` from testing-boss in SKILL.md and `flaky-detection.md`. (addresses B2, B3)

4. **[MEDIUM] Drop Python 3.9 from the classifier matrix** — Python 3.9 reached EOL October 2025. `pyproject.toml:20` still lists it. Remove `"3.9"` from `classifiers` and from `.github/workflows/ci.yml:13`. Update `requires-python = ">=3.10"`. This removes a CI leg that tests a dead Python and simplifies future syntax decisions. (addresses the 3.9 maintenance burden even if A6 currently passes)

5. **[MEDIUM] Add a real-execution integration test** — Add one test in `tests/test_integration.py` that pipes a real log file (from `tests/fixtures/logs/`) through the full `ci-guard watch --once` path using a subprocess call against a `gh`-stubbed environment or recorded fixture stream. Validates the `classify → ledger update` chain without live gh credentials. (addresses A2)

6. **[MEDIUM] Add `coverage` reporting to CI** — Add `coverage run -m unittest discover tests && coverage report` to `.github/workflows/ci.yml`. No threshold gate — coverage as a flashlight per Iron Law #5. Report as a PR comment or job summary. (addresses A3)

7. **[MEDIUM] Add cross-category negative companion tests** — Add at least one fixture-level negative per category pair (e.g., "known infra log must NOT classify as branch_failure", "known branch_failure log must NOT classify as test_flake"). (addresses A8)

8. **[LOW] Extend fixture corpus** — Add to `tests/fixtures/logs/`: a truncated log (last 200 bytes only), a log with ANSI color codes, a log with non-UTF8 bytes, a mixed-signal log with both branch_failure and infra_flake patterns. (addresses A10)

9. **[LOW] Add direct tests for `config.py`** — `config.py` contains `load_config()` and `check_script_freshness()` — pure or near-pure functions. A single `tests/test_config.py` would close the untested-module gap with minimal friction. (addresses A9, partially)

10. **[LOW] Link ci-guard → testing-boss in SKILL.md** — In SKILL.md Gate 2 (`branch_failure → patch code`), add a cross-reference: "When fixing the failing test, apply testing-boss's Iron Law #2 (push the test to the lowest layer that can detect the failure)." This closes the doctrine bridge gap. (addresses B6)
