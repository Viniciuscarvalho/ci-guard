## What to do

1. **Reproduce the flake locally.** Run the test in a tight loop:

   ```bash
   # pytest example
   for i in $(seq 1 50); do pytest -x {test_id} || break; done
   ```

   Most flakes reproduce within 50 iterations on a sufficiently loaded machine.

2. **Identify the cause.** Common patterns:
   - Implicit ordering assumption (depends on dict/set iteration order, file system listing order, etc.).
   - Time-of-day or timezone sensitivity.
   - Network call without retry logic against an external service.
   - Shared mutable state across tests (test pollution).
   - Race condition in the system under test (in which case the test is doing its job — the SUT is the bug).

3. **Fix the root cause, not the symptom.** Adding `@retry(times=3)` to a flaky
   test almost always hides a real defect. If a retry decorator is genuinely
   appropriate (e.g., the test exercises an inherently retried external call),
   document why in the test itself.

4. **Un-quarantine.** Remove the skip pragma and run:
   ```bash
   ci-guard ledger set-status \
       --test "{test_id}" --status fixed
   ```
   The ledger will keep observing it; if it flakes again, it'll come back.
